## Feature 1: Buffet Pre-Cost Estimation

**Phase:** 1 extension (weeks 2–3)
**Operator insight:** Most chefs don't re-cost recurring buffets. They prepare by instinct, scramble when under-prepped, overspend when over-prepped. Making the cost visible at setup is the single highest-leverage behavioral nudge in the system.

### How It Works

When a Buffet Set scan completes, the existing recognition pipeline already identifies items and their quantities. This feature adds the math + display:

```
total_set_cost = Σ (item.quantity × recipe.unit_cost_usd)
```

Displayed with:
- Total cost
- Breakdown by category (protein, pastry, produce, beverages, dairy, sundries)
- Comparison to 4-week historical average for same event type
- Per-cover estimate: `total_set_cost / expected_covers`

### The Nudge Behavior

If setup cost is >20% above what the Prophet forecast suggests is needed:
- Echo AI³ banner: *"This setup covers ~175 guests at normal consumption. Expected 140 — consider reducing bacon and pastries by one pan each?"*
- Chef can accept or dismiss; dismissals logged for pattern analysis

### Required Changes

**Database:** New columns on `waste.buffet_sessions`:
- `set_cost_usd numeric(12,2)` — total setup cost
- `set_cost_breakdown jsonb` — by category
- `estimated_covers_at_setup int` — what the cost covers at normal consumption
- `forecast_covers int` — from Prophet

**API:** Already covered by existing `POST /capture/buffet/set` response — extend to include cost rollup.

**UI:** Post-set review screen shows cost card (see wireframe `09-buffet-cost-view.md`).

### Why Phase 1

All required data (recognition, recipe costs, Prophet forecast) already exists. The feature is math + one UI card. 2 days of work.

## Feature 2: Cost-Per-Cover Calculation

**Phase:** 1 extension (weeks 2–3)
**Operator insight:** Hotel F&B directors want true food cost per buffet cover, broken out by consumed vs. wasted. It turns buffet from gut-feel operation into unit-economics operation.

### How It Works

Extends the Buffet Close flow with one new input (guest count) and produces the cost-per-cover breakdown:

```
consumed_cost = set_cost + refill_cost − close_cost
cost_per_cover_consumed = consumed_cost / guest_count
cost_per_cover_wasted = close_cost / guest_count
cost_per_cover_total = (set_cost + refill_cost) / guest_count
waste_pct = close_cost / (set_cost + refill_cost) × 100
```

### Guest Count Input

Three ways, in order of preference:
1. **POS integration** (Phase 2+): auto-pull from POS daypart total
2. **Group block + manual adjustment** (Phase 1): system pre-fills from reservations, chef confirms
3. **Manual entry** (Phase 1 fallback): chef types the number

Required field. Cannot close the buffet session without it.

### What The Chef Sees

See wireframe `09-buffet-cost-view.md` — the close screen adds a "cost per cover" summary card, styled like a receipt.

### Required Changes

**Database:** Additional columns on `waste.buffet_sessions`:
- `guest_count int`
- `guest_count_source text CHECK IN ('manual', 'pos', 'group_block_plus_manual')`
- `refill_cost_usd numeric(12,2)` — computed from refill entries (Phase 2, 0 in Phase 1)
- `close_cost_usd numeric(12,2)`
- `cost_per_cover_consumed_usd numeric(10,4)`
- `cost_per_cover_wasted_usd numeric(10,4)`
- `waste_pct numeric(5,2)`

**API:** `POST /buffet-sessions/:id/guest-count` — sets guest count after close scan, computes breakdown
       `GET /buffet-sessions/:id/cost-breakdown` — returns the full cost waterfall

### Phase 1 Caveat

Without refill tracking (Feature 4), the math assumes `refill_cost_usd = 0`. This is wrong in reality — most buffets get refilled. Document this clearly in the UI: *"Cost per cover assumes no refills. Enable refill tracking in Phase 2 for full accuracy."*

If the chef knows they refilled but can't itemize, allow a single number entry: "Estimated refills: $____" — logged with `refill_source: 'estimated'`.

### Why Phase 1

Math is simple. One new input. Huge operator value. Worth the scope add.

## Feature 3: Plus-Sign Multi-Zone Scanning

**Phase:** 1 extension (weeks 2–3)
**Operator insight:** A single capture session often covers multiple distinct zones. A pastry case has 6 shelves. A prep line has 3 workstations. Current UX forces separate captures or a blob. Neither is right.

### How It Works

During any capture mode, the chef can tap **[+] Add area** to create a new zone within the same capture session:

1. Chef scans first zone → reviews detected items
2. Taps [+] Add area → names zone (optional) → scans
3. Repeats N times
4. Taps Submit → all zones submitted as one entry with N zone sub-records

### Zone Naming

- **User-provided:** Free text, 40 char max (e.g. "Station 3", "Top shelf", "Cold line")
- **Auto-default:** If no name provided: `{outlet_code}-{station_code}-{YYYYMMDD}-{HHMM}-{zone_num}`
  - Example: `AG-S3-20260422-1018-02` = Aurora Grille Station 3, April 22 2026 at 10:18, zone 2
  - Deterministic, sortable, unique

### Why This Matters

- Reduces 4 separate captures to 1 captured session (4× less chef time)
- Preserves zone-level detail for analytics (which shelf has the most pastry waste)
- Natural mental model — chefs already think in terms of zones

### Required Changes

**Database:** New table `waste.capture_zones` linked to `waste.entries` (see migration 012).

**API:** `POST /capture/*` endpoints accept optional `zones[]` parameter. Response shape gains `zones[]` array.

**UI:** Capture flow gains an "Add area" button on the review screen (see wireframe `10-multi-zone-scan.md`).

### Why Phase 1

Small additive feature. Chef adoption is obvious. Worth building early so analytics has zone-level data from day one.

## Feature 4: Refill Tracking via Prep-Line Model

**Phase:** 2 (weeks 4–6)
**Operator insight:** Refills are the hard-to-keep number. Without them, buffet math is wrong.

### Why NOT a "refill button" at the buffet

We considered: "+ Refill" button that chef taps at the buffet when topping up. Rejected because:
- Chef is busy during service — this is the #1 thing that gets skipped
- No additional data captured (just an increment)
- Breaks the "scan once, done" model
- Habit won't survive real kitchens

### Why prep-line tracking works

The insight: **chefs don't refill the buffet — they move product from the prep line to the buffet.** Capture the event where the chef's attention already is.

**Workflow:**
1. Chef preps a fresh pan of eggs at 8:15
2. In the app, hits "Log refill" (or says "Log refill for station 3" to Echo)
3. Scans the fresh pan (standard video scan, 5 seconds)
4. System logs: pan of eggs, 6000g, $24 added to Station 3 at 8:15
5. Chef walks the pan to the buffet

**Data captured:**
- Refill quantity (exact, not estimated)
- Refill cost (from recipe)
- Prep-to-buffet timing (useful for operational analysis)
- Doubles as prep yield tracking

### Mid-Service Scan Inference

If the chef skips logging but does a mid-service scan:
- System compares to set + mid scan delta
- Infers: "Station 3 eggs: set 40 portions, mid-scan 35. Either 5 consumed or refills happened."
- Asks: "Did you refill Station 3 eggs since set?" → Yes/No/Can't remember
- If Yes → system logs an estimated refill with `refill_source: 'inferred'` and flags for review

### Required Changes

**Database:** New columns on `waste.entries`:
- `buffet_session_id uuid` (FK to `waste.buffet_sessions`) — links refill captures to the buffet session
- `refill_source text CHECK IN ('prep_line', 'inferred', 'estimated', null)` — null for non-refill captures

**New capture mode:** `buffet_refill` added to `capture_mode` CHECK constraint.

**API:** `POST /capture/buffet/refill` — new endpoint (like `buffet_mid` but logged as refill)

**UI:** "Log refill" button appears on home screen when an active buffet session exists for the user's station. See wireframe (add to `05-buffet-set-close.md` in a future supplement).

### Why Phase 2

Requires chef adoption of a new habit. Budget extra UX polish time. Best tested when Phase 1 MVP is stable and chefs are already using the app.

## Feature 5: Menu Pre-Load (3 Input Modes)

**Phase:** 2 (weeks 4–6)
**Operator insight:** Even with EchoLayout, not every event has perfect structured data. Catering, off-sites, one-offs, new items — the menu exists on paper or in a Google Doc. Menu pre-load bypasses that.

### Three Input Modes

All three produce the same output: an event-scoped recipe shortlist that persists for the service.

#### Mode A: Menu Screenshot
- Chef uploads an image (digital menu PDF page, POS screenshot, Google Doc export)
- Echo AI³ runs OCR + vision analysis (`menu-extraction` prompt)
- Extracts dish names, confidence-scored
- Fuzzy-matches to recipe database (trigram + LLM re-rank)

#### Mode B: Printed Menu Photo
- Chef snaps the paper menu on their counter
- Same pipeline as Mode A
- Lower OCR accuracy — expect more chef confirmation on matches

#### Mode C: Verbal Menu Rundown
- Chef says: *"Today we've got chicken piccata, eggplant parm, salmon en croute, and the soup is minestrone"*
- Transcribe → parse into dish list → match to recipes
- Same output shape as A and B

### Output: Event Menu

All three modes write to a new `waste.event_menu` table:

```
event_menu {
  event_id,
  menu_source: 'screenshot' | 'photo' | 'verbal',
  extracted_items: [
    { dish_name, recipe_id, confidence, status: 'matched' | 'unmatched' | 'pending_chef_review' }
  ],
  parsed_at,
  parsed_by_user_id,
  raw_input_url   // the screenshot, photo, or audio
}
```

During the event, every capture within that event uses `extracted_items` as the station shortlist (taking precedence over EchoLayout defaults when scoped to the event).

### Unmatched Items

When extraction can't find a recipe match:
- Flag as `unmatched`
- Chef is prompted once: "New dish — Carbonara? Want to create a draft recipe?"
- Accepting routes to Feature 6 (verbal recipe creation)

### Required Changes

**Database:** `waste.event_menu` table — see migration 013 (stub in Phase 1, populated in Phase 2).

**API:**
- `POST /events/:event_id/menu/upload` — accepts image or audio, returns extracted items
- `GET /events/:event_id/menu` — returns current event menu
- `PATCH /events/:event_id/menu/items/:item_id` — chef corrects a match

**Prompts:** `menu-extraction.md` (see `04-prompts-new/`).

### Why Phase 2

Requires OCR pipeline and extraction prompt work. Needs integration testing with recipe fuzzy match. Not blocking Phase 1 shipping.

## Feature 6: Verbal Recipe Creation (Draft-Only)

**Phase:** 2 (weeks 4–6) — with strict scoping
**Operator insight:** Kitchen creativity outpaces admin work. New dishes get cooked, served, and charged before anyone computes food cost. This feature means the plate is costed the moment it's conceived.

### The Critical Constraint: Draft-Only

This feature **creates `draft: true` recipes that do not flow into live inventory, menu costing, or P&L until a chef reviews and approves them.** Auto-writing AI-generated recipes to production accounting is dangerous — a bad parse becomes a bad costing which becomes bad margin reporting. Draft mode eliminates that risk.

### Workflow

1. Chef plates a new dish, takes a video of the plate
2. Records verbal description:
   > *"6oz Manhattan strip, sautéed broccoli with oil garlic red pepper flakes, pomme purée, demi-glace."*
3. Echo AI³ transcribes audio
4. Claude parses transcript via `verbal-recipe-extraction` prompt into structured components:
   ```json
   {
     "dish_name": "Manhattan Strip with Broccoli and Pomme Purée",
     "components": [
       { "type": "protein", "name": "Manhattan strip", "quantity": 6, "unit": "oz", "prep": "seared" },
       { "type": "vegetable", "name": "broccoli", "quantity": null, "unit": "portion", "prep": "sautéed",
         "sub_ingredients": ["olive oil", "garlic", "red pepper flakes"] },
       { "type": "starch", "name": "pomme purée", "quantity": null, "unit": "portion" },
       { "type": "sauce", "name": "demi-glace", "quantity": null, "unit": "portion" }
     ]
   }
   ```
5. For each component, fuzzy-match to existing culinary module items (stocks, preps, ingredients, sub-recipes)
6. Unmatched components flagged — chef can voice those too or defer
7. System creates a **draft** recipe in culinary module (`draft: true`)
8. Computes estimated plate cost from matched components
9. Video attached as plate reference image
10. Chef sees draft review screen: cost estimate, matched/unmatched components, edit controls
11. Chef hits "Promote to live" when ready — recipe becomes production

### What Happens Downstream

Once promoted:
- Recipe is a full citizen in the culinary module
- Flows into menu engineering, plate builder, P&L
- Visible to EchoWaste recognition (and eventually EchoLayout for menu assembly)
- Attached video is available as training data for future plate recognition

### Quality Expectations

Verbal recipe parsing has a quality ceiling. Even with Claude:
- Common phrasings parse at ~85%+ accuracy
- Non-standard abbreviations, ad-hoc weights, brand-name shorthand drop accuracy
- Edge cases ("about 6 oz of whatever New York we got in today") fail gracefully — flag as unclear, prompt chef

Set expectations: *"I drafted this — review before shipping."* Never auto-promote.

### Required Changes

**Database:**
- Requires `draft boolean DEFAULT false` column on `culinary.recipes` — **coordinate with culinary module team**
- New table `waste.draft_recipes` tracks EchoWaste-originated drafts:
  - `id, recipe_id, created_from_entry_id, parsed_components jsonb, matched_components jsonb, unmatched_components jsonb, estimated_cost_usd, video_url, created_at, promoted_at, promoted_by`

**API:**
- `POST /draft-recipes` — accepts video + audio, runs extraction, creates draft
- `GET /draft-recipes/:id` — full draft with matched/unmatched components
- `PATCH /draft-recipes/:id` — chef edits components before promote
- `POST /draft-recipes/:id/promote` — promotes to live recipe, flips `culinary.recipes.draft = false`

**Prompts:** `verbal-recipe-extraction.md` (see `04-prompts-new/`).

### Why Phase 2 (Not Phase 3)

The feature has clear high-value. The draft-only scoping removes the risk. Shipping in Phase 2 means chefs can start generating training data for Phase 3 auto-matching improvements.

### What Stays In Phase 3

- Automatic nutritional calculation
- Allergen tagging
- Full plating instructions with component positioning (requires much richer vision work)
- Auto-promotion with confidence threshold
- Cross-property draft sharing (Network Intelligence)

## Cross-Feature Interactions

These features compound in intentional ways:

1. **Feature 1 + Feature 4:** The set cost + refill cost feeds the real-time cost-per-cover. As refills happen, the number updates live. Chef can see "we're $2.18/cover so far, trending to $2.41 at close."

2. **Feature 3 + Feature 1:** Multi-zone scanning lets the Set scan cover the entire buffet in one capture, with zones = stations. Per-station cost rollup falls out automatically.

3. **Feature 5 + Feature 6:** When menu pre-load encounters an unmatched dish, it offers to create a draft recipe. The chef voices the recipe, Feature 6 handles it, the draft appears in the event menu. Closed loop.

4. **Feature 2 + Guardian AI:** Cost-per-cover anomalies (today's brunch was $4.12/cover vs. $2.80 usual) trigger Guardian AI investigation automatically.

5. **Feature 6 + EchoLayout:** Draft recipes promoted from verbal creation become available in EchoLayout for future event planning. The chef's on-the-fly creativity feeds the future structured plan.
