D11 · Unified schedule view (AutoScheduling + Schedule merge)

  Echo's scheduling lives across two surfaces today:
    · Echo Schedule: live roster (manager's actual)
    · AutoScheduling (chronos_flex_labor + orchestrator):
      AI's per-station headcount proposal

  The chef should see ONE schedule with proposed-vs-actual side
  by side, not two screens. This module is the reconciliation
  layer — does NOT remove either underlying surface (each is
  still source of truth for its own write path); reads from both
  and produces a unified payload.

  Endpoints (/api/schedule/unified)

    GET /{outlet_id}?date=YYYY-MM-DD
        Returns proposal + actual + per-station variance with
        status (match | over | under).

    POST /{outlet_id}/accept-proposal?date=
        Promote selected proposal stations into actual shift
        rows via the existing echo_schedule write path.
        Idempotent: stations already at >= proposed_headcount
        skip; gap-filling only.

  Doctrine alignment

    · §1.4: operator audience required.
    · §3.1: accept-proposal writes new shift rows; never edits
      historical rows.
    · D27 tenant isolation on every read/write.

  What this enables

    Friday 7am, chef opens unified view:
      GET /api/schedule/unified/signature_room?date=2026-05-09
        → grill: proposed 4 / actual 4 (match)
        → sauce: proposed 3 / actual 2 (under by 1)
        → expo: proposed 2 / actual 3 (over by 1)
    Chef taps "Accept proposal · sauce" — system fires +1 shift
    row for sauce. Manager fills the assignment. Done in 3 taps.
